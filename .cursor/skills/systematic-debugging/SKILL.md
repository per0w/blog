# Систематический дебаг

## Железное правило

```
НИКАКИХ ИСПРАВЛЕНИЙ БЕЗ ИССЛЕДОВАНИЯ КОРНЕВОЙ ПРИЧИНЫ
```

Если фаза 1 не завершена — нельзя предлагать фикс.
Исправление симптома — это провал, а не решение.

## Когда использовать

Использовать при **любой** технической проблеме:

- Ошибки в консоли / терминале
- Неожиданное поведение UI
- Проблемы с синхронизацией / offline-режимом
- Ошибки сборки или TypeScript
- Проблемы с Prisma / миграциями
- Регрессии после рефакторинга

**Особенно** когда:

- Хочется «просто быстро починить» — это сигнал остановиться
- Уже попробовал 1-2 фикса и они не помогли
- Не до конца понимаешь, почему сломалось
- Проблема проявляется нестабильно

## Четыре фазы

Каждая фаза завершается до перехода к следующей.

### Фаза 1: Исследование корневой причины

**До любых попыток исправления:**

1. **Прочитать ошибку целиком**
   - Полный stack trace, не пропуская строки
   - Номера строк, пути к файлам, коды ошибок
   - Сообщения в консоли браузера и терминале

2. **Воспроизвести стабильно**
   - Точные шаги для повторения
   - Воспроизводится ли каждый раз?
   - Если нет — собрать больше данных, не гадать

3. **Проверить недавние изменения**
   - `git diff` — что изменилось?
   - Новые зависимости, изменения конфига
   - Изменения в Prisma-схеме, миграции

4. **Проследить поток данных**
   - Откуда приходит некорректное значение?
   - Трассировать назад по цепочке вызовов до источника
   - Исправлять в источнике, а не в месте проявления

#### Trenika-специфичные проверки

- **Offline-режим**: проблема только в offline или и в online тоже?
- **IndexedDB**: проверить состояние данных в DevTools → Application → IndexedDB
- **Sync queue**: есть ли зависшие операции в очереди синхронизации?
- **Service Worker**: не отдаёт ли SW закешированную версию? Попробовать с отключённым SW
- **Prisma**: совпадает ли схема с миграциями? `npx prisma migrate status`

### Фаза 2: Анализ паттерна

1. **Найти рабочий аналог**
   - Есть ли похожий работающий код в проекте?
   - Что отличается между рабочим и сломанным?

2. **Сверить с контрактами**
   - DTO в `packages/shared` соответствуют ожиданиям?
   - API-ответ совпадает с типом?
   - Маппер корректно преобразует данные?

3. **Проверить границы слоёв**
   - route → handler → service → repository — на каком слое ломается?
   - Frontend: page → widget → feature → entity → shared — где проблема?

### Фаза 3: Гипотеза и тест

1. **Сформулировать одну гипотезу**
   - Чётко: «Я считаю, что X — корневая причина, потому что Y»
   - Записать прежде чем действовать

2. **Проверить минимально**
   - Наименьшее возможное изменение для проверки гипотезы
   - Одна переменная за раз
   - Не исправлять несколько вещей одновременно

3. **Оценить результат**
   - Подтвердилось? → Фаза 4
   - Не подтвердилось? → Новая гипотеза, вернуться к фазе 1 с новой информацией

4. **Если не знаешь**
   - Сказать «я не понимаю X»
   - Не притворяться, что понимаешь

### Фаза 4: Исправление

1. **Исправить корневую причину**
   - Одно изменение за раз
   - Никаких «раз уж я тут» улучшений
   - Никакого рефакторинга в одном коммите с фиксом

2. **Верифицировать**
   - Проблема действительно решена?
   - Ничего другого не сломалось?
   - `tsc --noEmit` проходит?
   - Приложение собирается?

3. **Правило трёх попыток**
   - Если 3+ фикса не помогли → **остановиться**
   - Это сигнал архитектурной проблемы, а не бага
   - Обсудить с пользователем, прежде чем продолжать

## Красные флаги — СТОП

Если ловишь себя на мысли:

- «Сейчас быстро починю, потом разберусь»
- «Просто попробую поменять X и посмотрю»
- «Добавлю несколько изменений, запущу тесты»
- «Наверное это X, давай починю»
- «Я не до конца понимаю, но может сработать»

**Все эти мысли = СТОП. Вернуться к фазе 1.**

## Распространённые отговорки

| Отговорка                                 | Реальность                                                                       |
| ----------------------------------------- | -------------------------------------------------------------------------------- |
| «Баг простой, процесс не нужен»           | У простых багов тоже есть корневые причины. Процесс быстрый для простых случаев. |
| «Срочно, нет времени на процесс»          | Систематический подход быстрее, чем хаотичное угадывание.                        |
| «Сначала попробую, потом расследую»       | Первый фикс задаёт паттерн. Начни правильно.                                     |
| «Несколько фиксов за раз сэкономят время» | Невозможно понять, что сработало. Создаёт новые баги.                            |
| «Ещё одна попытка» (после 2+)             | 3+ провалов = архитектурная проблема.                                            |

## Краткая шпаргалка

| Фаза                | Что делать                                                         | Критерий завершения          |
| ------------------- | ------------------------------------------------------------------ | ---------------------------- |
| 1. Корневая причина | Прочитать ошибку, воспроизвести, проверить diff, проследить данные | Понимаю ЧТО и ПОЧЕМУ         |
| 2. Паттерн          | Найти рабочий аналог, сверить контракты, проверить слои            | Нашёл отличие                |
| 3. Гипотеза         | Сформулировать, проверить минимально                               | Подтверждена или новая       |
| 4. Исправление      | Один фикс, верификация                                             | Баг решён, ничего не сломано |
