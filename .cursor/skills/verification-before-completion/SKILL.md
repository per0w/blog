# Верификация перед завершением

## Железное правило

```
НИКАКИХ ЗАЯВЛЕНИЙ О ГОТОВНОСТИ БЕЗ СВЕЖИХ ДОКАЗАТЕЛЬСТВ
```

Если команда верификации не запущена в этом сообщении — нельзя заявлять, что всё работает.

## Когда применять

**ВСЕГДА перед:**

- Любым заявлением о готовности («сделано», «готово», «исправлено», «работает»)
- Любым выражением удовлетворённости («отлично», «идеально»)
- Коммитом или созданием PR
- Переходом к следующей задаче
- Завершением сессии

## Чеклист верификации

Запускать команды **по порядку**. Каждая должна пройти перед следующей.

### 1. TypeScript — нет ошибок типов

```bash
pnpm typecheck
```

Ожидаемый результат: нет ошибок. Если ошибки есть — исправить до продолжения.

### 2. Lint — нет ошибок линтера

```bash
pnpm lint
```

Ожидаемый результат: нет ошибок (предупреждения допустимы). При ошибках:

```bash
pnpm lint:fix
```

Если `--fix` не справился — исправить вручную.

### 3. Форматирование — код отформатирован

```bash
pnpm format:check
```

Ожидаемый результат: все файлы отформатированы. Если нет:

```bash
pnpm format
```

### 4. Сборка — проект собирается

```bash
pnpm build
```

Ожидаемый результат: exit code 0, без ошибок.

### 5. Smoke-чеклист (ручной — описать пользователю)

Поскольку автотесты пока отложены (правило `07-delivery-no-tests-yet`), обязательно описать пользователю:

- **Happy path**: что проверить, чтобы убедиться, что фича работает
- **Ошибки валидации**: какие невалидные данные попробовать
- **Offline/online**: если фича связана с данными — проверить с отключённой сетью
- **Reload**: сохраняются ли данные после перезагрузки страницы
- **Откат**: что откатывать, если после применения обнаружится проблема

## Формат доказательства

Каждое заявление о готовности должно сопровождаться фактом:

| Заявление          | Требуется                                                    | Недостаточно                       |
| ------------------ | ------------------------------------------------------------ | ---------------------------------- |
| «TypeScript чисто» | Вывод `pnpm typecheck`: 0 ошибок                             | «Должно работать»                  |
| «Lint чисто»       | Вывод `pnpm lint`: 0 ошибок                                  | «Я проверил глазами»               |
| «Собирается»       | Вывод `pnpm build`: exit 0                                   | «Lint прошёл» (lint ≠ сборка)      |
| «Баг исправлен»    | Воспроизведение оригинальных шагов → проблема не повторяется | «Код изменён»                      |
| «Фича готова»      | Smoke-чеклист описан                                         | «Тесты проходят» (тестов пока нет) |

## Красные флаги — СТОП

Если ловишь себя на словах:

- «Должно работать»
- «Скорее всего всё ок»
- «Я уверен»
- «Выглядит правильно»
- «Сделано!» (без запуска проверок)
- «Готово» (на основе прошлого запуска)

**ВСЕ эти фразы = СТОП. Запустить чеклист.**

## Распространённые отговорки

| Отговорка                       | Реальность                     |
| ------------------------------- | ------------------------------ |
| «Должно работать»               | ЗАПУСТИ проверку               |
| «Я уверен»                      | Уверенность ≠ доказательство   |
| «Только в этот раз»             | Исключений нет                 |
| «Lint прошёл»                   | Lint ≠ компилятор ≠ сборка     |
| «Я устал»                       | Усталость ≠ оправдание         |
| «Частичная проверка достаточна» | Частичная не доказывает ничего |

## Минимальная проверка

Если времени совсем мало — **абсолютный минимум**:

```bash
pnpm typecheck && pnpm lint && pnpm build
```

Одна команда, три проверки. Если хоть одна упала — не заявлять готовность.

## Связь с другими правилами

- **Правило 07 (delivery-no-tests-yet)**: этот скилл компенсирует отсутствие автотестов строгой ручной верификацией
- **Правило 12 (execution-discipline)**: smoke-чеклист из этого скилла = часть завершения сессии
- **Скилл systematic-debugging**: если верификация провалилась — использовать дебаг-скилл для поиска причины, а не гадать
