Раньше часто прятали блок через `isOpen && <Panel />` — и теряли состояние, скролл и введённый текст. В React 19.2 для этого появился отдельный примитив: **`<Activity>`** — прятать интерфейс «по-взрослому», не размонтируя дерево.

![Иллюстрация к заметке про Activity в React](/images/blog/react-activity-og.jpg)

## Что это

`<Activity>` переводит дочерний UI в один из режимов:

- **`visible`** — обычная работа, как сейчас.
- **`hidden`** — элемент скрыт (визуально, в том числе через `display: none`), **эффекты снимаются**, нагрузка на UI снижается, при этом **DOM и состояние сохраняются**.

Скрытые ветки могут продолжать обновляться с **пониженным приоритетом** — это часть модели «не держать пользователя, но не терять готовый UI».

Подробности и контекст релиза — в [заметке команды React про View Transitions, Activity и эксперименты](https://react.dev/blog/2025/04/23/react-labs-view-transitions-activity-and-more#activity).

## Зачем на практике

- Скрыть вкладку или боковую панель **без сброса формы и скролла**.
- **Предзагрузить** следующий экран (в связке со `Suspense`).
- Не гонять тяжёлый UI через полный размонт/монт при каждом переключении.
- Поставить медиа на паузу в `cleanup`, не теряя позицию при возврате.

Раньше оставались костыли: только CSS, полный размонт или ручное сохранение состояния снаружи. `<Activity mode="hidden">` задаёт один явный контракт.

## Примеры

**Сохранить состояние панели:**

```tsx
<Activity mode={isOpen ? "visible" : "hidden"}>
  <Sidebar />
</Activity>
```

**Подогреть тяжёлую вкладку:**

```tsx
<Suspense fallback={<Spinner />}>
  <Activity mode="hidden">
    <HeavyTab />
  </Activity>
</Suspense>
```

**Табы без сброса локального state:**

```tsx
<Activity mode={tab === "A" ? "visible" : "hidden"}>
  <TabA />
</Activity>
<Activity mode={tab === "B" ? "visible" : "hidden"}>
  <TabB />
</Activity>
```

## Осторожно с эффектами

Для эффектов режим **`hidden` близок к размонтированию**: подписки, сокеты, таймеры и воспроизведение медиа нужно **корректно чистить в `cleanup`**, иначе утечки и фоновая работа останутся сюрпризом.

```tsx
useEffect(() => {
  ref.current?.play();
  return () => ref.current?.pause();
}, []);
```

При скрытии — пауза и остановка побочек; при возврате — можно продолжить с того же места, если состояние плеера живёт в ref/DOM.

## Дашборд с графиками

Типичная боль: десятки графиков с частым обновлением данных и переключение вкладок.

- **Только `display: none` в CSS** — визуально тихо, но эффекты и лишние обновления могут продолжать жечь CPU: FPS падает.
- **Полный размонт** — производительность в скрытом режиме лучше, но при возврате всё пересоздаётся, теряется «тёплое» состояние.

С `<Activity>` скрытые графики **отключают эффекты и лишнюю работу**, но **DOM и state остаются** — возврат на вкладку быстрее и предсказуемее для UX.

## Замеры (иллюстрация, не бенчмарк)

Справедливый вопрос: раз визуально тоже используется `display: none`, в чём выигрыш? **В управлении эффектами и работой React**, а не в «магии» CSS.

Ориентировочные цифры с синтетического стенда (React 19.2, Vite, Chrome; дашборд с ~20 графиками, обновление ~каждые 16 мс; среднее по 10 прогонам, от клика по вкладке до запуска эффектов внутри компонента):

| Подход                                   | Поведение                                                                             |
| ---------------------------------------- | ------------------------------------------------------------------------------------- |
| CSS `display: none`                      | ~480 мс — скрыто визуально, фоновые эффекты/обновления продолжают нагружать движок.   |
| `<Activity>`, первый показ после скрытия | ~517 мс — дороже за счёт переподключения эффектов.                                    |
| `<Activity>`, повторный показ            | ~188 мс — DOM уже готов, пока вкладка была скрыта, браузер не тратил ресурсы впустую. |

Это **не универсальный бенчмарк**: на реальном проекте цифры будут другими. Вывод практический: для сценариев «живые графики + частые переключения» выгодно **не кормить скрытые ветки эффектами**, но **не платить полным пересозданием DOM** при каждом возврате.

## Вывод

`<Activity>` — ещё один рычаг контроля над UI:

- отложить и смягчить работу тяжёлых блоков;
- прятать без потери состояния;
- держать перформанс ближе к размонтированию по эффектам, но с быстрым «возвратом».

**Практично:** прячь то, куда пользователь реально вернётся в ближайшее время; не копи бесконечные `hidden`-деревья «на всякий случай».
